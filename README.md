# Instructions

## Setup

1. Install dependencies from the repo root:
   ```bash
   uv sync
   ```
2. Create a `.env` file in this directory with your OpenRouter key:
   ```
   OPENSOURCE_KEY=your_key_here
   ```

## Problem Generation

Generate problems and append them to a JSONL file:
```bash
# Basic (no noise modifiers)
uv run python -m constraints.generate --num-samples 5 --num-names 10 --output constraints/new_problems.jsonl

# With difficulty modifiers (combine any flags)
uv run python -m constraints.generate --num-samples 5 --num-names 10 --synonyms --negations --adverbs --connectives --output constraints/new_problems.jsonl

# Generate every flag combination for n=2..MAX_NUM names
uv run python -m constraints.generate --complete-problems MAX_NUM --output constraints/complete_problems.jsonl
```

## Problem Evaluation

Run all problems against the model:
```bash
uv run python -m constraints.evaluate --problems constraints/new_problems.jsonl --output constraints/new_results.jsonl --num-runs 10
```

Evaluate a single problem by index (0-based):
```bash
uv run python -m constraints.evaluate --problems constraints/problems.jsonl --problem-index 2 --num-runs 3
```

# Write-up

## 1. Task Selection
The task I've decided to implement is a simple constraint problem where an LLM has to order a list of people by their height.

For example given an ascending (by height) list of people's names e.g. ['Alice', 'Bob', 'Rachel']. My problem generator transforms this into text e.g. "Alice is shorter than Bob. Rachel is taller than Bob."

It ensures that there is only one correct solution by making sure that each adjacent pair is described in the prompt for the LLM [Alice, Bob] and [Bob, Rachel].

LLMs can handle this ok with a few names however struggle when:
- Many constraints (lots of names)
- Synonyms (Alice is smaller than Bob)
- Negation (Alice is not as tall as Bob)
- Connectives (and, but)
- Adverbs (slightly taller, much taller)

## 2. Model
`openai/gpt-4.1-mini` via OpenRouter. Chosen because it handles small problems well but degrades 
meaningfully as complexity increases, making it ideal for stress-testing.

## 3. Data Generation Strategy
Problems are generated by:
1. Sampling N random names and defining a ground-truth ordering
2. Generating adjacent-pair comparisons (guaranteeing a unique solution)
3. Optionally applying noise: synonyms, negations, adverbs, connectives

This is scalable — `--complete-problems` generates all 2^4 flag combinations for every list size given a maximum number
from 2 to N, producing [1568] problems for downstream analysis (if MAX_NUM = 99).

## 4. Verification
`verify.py` parses the model's output with a regex to extract a Python list, normalises case, 
and compares it element-by-element to the golden label.

## 5. Problems
For each problem, I use 15 names (I've written the pass rate against each problem with 10 tries each).
- 1. Easy problem: no special parameters (30%)
- 2. Synonyms Only (20%)
- 3. Adverbs Only (40% - better than without (1) apparently)
- 4. Negations Only (50% - even better!)
- 5. Connectives Only (40%)
- 6. All of the above (60% - GPT really locked in for this one)
Overall it had a 40% success rate.


These problems can be found in problems.jsonl Results can be found in results.jsonl

## 6. Bonus
I've also attached a file complete_problems.jsonl. This file includes every combination of parameter from number of
names = 2 to 99. This data can be then used for future analysis to see which parameters the LLM struggles with processing. There are 1,568 problems total.

It would be great to plot this data on to a graph to see which kind of paramaters the LLM is most confident using, and at what value of N (names) the LLM really begins to struggle.
